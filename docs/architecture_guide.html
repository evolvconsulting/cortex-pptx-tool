What is a Custom Tool?

Understanding how Cortex Agents extend their capabilities

A Cortex Agent is an AI orchestrator that can use tools to take actions. Snowflake provides built-in tool types like cortex_analyst_text_to_sql, cortex_search, and web_search. But when you need the agent to do something entirely custom — like generating PowerPoint presentations — you build a Custom Tool.

A custom tool is a stored procedure that the agent can call. But behind that procedure can live an entire ecosystem: containerized services, external APIs, Python logic, and more.

The Core Concept

From the agent's perspective, a custom tool is simple: it calls a procedure with parameters and gets a result back. But the depth of what happens inside that procedure is where the real architecture lives.

This guide walks through every layer — from the smallest utility function inside a Docker container, all the way up to the main orchestrating agent that invokes the tool.

1The Service: Utilities & Converters

The smallest building blocks inside the SPCS container

Layer 1 · Foundation

At the very bottom of the architecture are pure utility functions that handle unit conversion between web standards (pixels, RGB) and PowerPoint's internal units (inches, points, hex colors).

Unit Conversions (The Atomic Level)

// src/config/constants.js
const CONSTANTS = {
  PT_PER_PX: 0.75,         // 1px = 0.75 points
  PX_PER_IN: 96,           // 96px = 1 inch
  EMU_PER_IN: 914400,      // EMU = PowerPoint's internal unit

  DEFAULT_WIDTH: 960,      // 10" x 96dpi
  DEFAULT_HEIGHT: 540,     // 5.625" x 96dpi (16:9)
  SLIDE_WIDTH_IN: 10,
  SLIDE_HEIGHT_IN: 5.625,
};

// src/utils/units.js
pxToInch(px)       // px / 96 → inches for PptxGenJS positioning
pxToPoints(pxStr)  // parseFloat(px) * 0.75 → font sizes
rgbToHex(rgbStr)   // "rgb(209,86,53)" → "D15635" (no # prefix!)

DOM Extraction (Playwright)

The domExtractor (~970 lines) is the core engine. It uses Playwright to render HTML in a headless Chromium browser, then walks the DOM tree extracting every visible element with its computed CSS properties:

// src/converters/domExtractor.js  extractSlideData(page)
// Returns structured data for each element:
{
  type: 'text',     // or 'shape', 'list', 'line', 'image'
  text: 'Revenue Growth',
  position: { x: 0.31, y: 0.31, w: 6.88, h: 0.52 },  // inches
  style: {
    fontSize: 28,      // points (converted from CSS px)
    fontFace: 'Arial',
    color: '2E3952',   // hex without #
    bold: true,
    italic: false,
    align: 'left'
  }
}

HTML-to-PPTX Conversion Pipeline

What Gets Extracted from the DOM?

  • Text elements (<p>, <h1>-<h6>) → font, size, color, bold, italic, alignment
  • Lists (<ul>, <ol>) → PptxGenJS bullet format with indentation
  • Shapes (<div> with background/border) → rounded rectangles with fill, shadow
  • Placeholders (class="placeholder") → reserved positions for charts/tables
  • Inline formatting (<b>, <i>, <span>) → text runs with mixed styles
  • Rotation (CSS transform: rotate()) → PptxGenJS rotation property

Three Conversion Strategies

Strict Converter

domParserConverter.js

Validates HTML strictly. Used when exactness matters. Rejects invalid structures.

Flexible Converter (Primary)

flexibleConverter.js

Tolerant, auto-scales overflowing content. This is the main conversion path used by the agent.

Screenshot Converter

screenshotConverter.js

Captures HTML as PNG image and embeds it. Fast but not editable.

Slide Builder

slideBuilder.js

Takes extracted DOM data and creates native PptxGenJS elements: addText(), addShape(), addImage().

Chart & Table Builders

Charts and tables are not rendered from HTML. Instead, the agent provides structured JSON data, and these builders create native PptxGenJS chart/table objects at placeholder positions:

// src/converters/chartBuilder.js
function addChartToSlide(slide, pres, chartConfig, placeholder) {
  slide.addChart(pres.charts[chartConfig.type], chartConfig.data, {
    x: placeholder.x,   // Position from HTML placeholder
    y: placeholder.y,
    w: placeholder.w,
    h: placeholder.h,
    ...chartConfig.options   // Colors, legends, axis titles
  });
}
// Supported: BAR, LINE, PIE, DOUGHNUT, SCATTER, AREA

1The Service: Express API

The HTTP interface that Snowflake calls

Layer 1 · Service API

All the utilities and converters are exposed through a simple Express.js HTTP server with three endpoints:

Endpoint Method Purpose
GET /health GET Health check with dependency status (Playwright, LibreOffice)
POST /convert POST Convert HTML slides to PPTX (the main path)

The /convert Pipeline

This is the endpoint that the Snowflake service function calls. Here's the full pipeline:

1
Receive Snowflake Service Function call
Input format: {"data": [[0, {"slides": [...], "options": {...}}]]}
2
For each slide: Write HTML to temp file, launch Playwright browser
3
DOM Extraction: Navigate Chromium to the HTML file, extract all elements with computed styles
4
Build Slide: Create PptxGenJS slide, add backgrounds, text, shapes, lists, images as native editable elements
5
Process Charts/Tables: Find placeholders by ID, inject native charts/tables at those positions
6
Return: {"output": {"base64": "UEsDBBQA...", "slideCount": 8}}

1The Service: Docker & SPCS

Containerizing the service for Snowpark Container Services

Layer 1 · Container

The Node.js service runs inside a Docker container deployed to Snowpark Container Services (SPCS). The Dockerfile uses a multi-stage build to minimize the final image size:

# Stage 1: Builder - compile native dependencies (sharp, etc.)
FROM node:18-slim AS builder
RUN apt-get install -y build-essential python3 libvips-dev
COPY package.json .
RUN npm ci --omit=dev

# Stage 2: Runtime - minimal production image
FROM node:18-slim
# Runtime dependencies only (no build tools):
#   libvips42        - image processing (sharp)
#   libreoffice      - PPTX-to-PDF preview
#   poppler-utils    - PDF-to-image (pdftoppm)
#   Chromium deps    - Playwright headless browser

COPY --from=builder /app/node_modules ./node_modules
RUN npx playwright install chromium
COPY src/ ./src/

EXPOSE 8080
CMD ["node", "src/index.js"]

Key Dependencies in the Container

Dependency Purpose
PptxGenJS Creates native .pptx files programmatically
Playwright + Chromium Renders HTML and extracts DOM with computed CSS
Sharp (libvips) Image processing (resize, format conversion)
LibreOffice Impress PPTX → PDF conversion for preview generation
poppler-utils PDF → PNG images (slide previews)
Resource Allocation: The SPCS service runs with 2G–4G RAM and 1 CPU. Chromium headless rendering is the most resource-intensive operation.

2Infrastructure

Image repo, stage, compute pool, SPCS service

Layer 2

Creates the foundational Snowflake objects:

  • Image Repository — Where you push the Docker image
  • Output Stage — Where generated PPTX files are stored
  • Compute Pool — Resources for the container
  • SPCS Service — The running container
  • Service Function — SQL interface to call the container

2Stored Procedure

Orchestrates service call and file output

Layer 2

Creates CREATE_PPTX_FROM_JSON — the procedure that:

  • Calls the SPCS service
  • Decodes the base64 response
  • Writes the PPTX file to the output stage

3PPTX Agent

Specialist Cortex Agent for presentations

Layer 3

Creates PPTX_GENERATOR_AGENT — a specialist Cortex Agent with:

  • Detailed system instructions for HTML slide design
  • 18 color palettes and 7 slide templates
  • The CREATE_PPTX_FROM_JSON tool bound
Standalone Usage: At this point you can use PPTX_GENERATOR_AGENT directly from Snowsight. Layer 4 is only needed for multi-agent orchestration.

4Multi-Agent Bridge

JWT authentication for agent-to-agent calls

Layer 4

Enables your main agent to call PPTX_GENERATOR_AGENT via JWT authentication. Creates:

  • Network Rule — Allows HTTPS to Snowflake APIs
  • Secret — Stores the RSA private key
  • External Access Integration — Bridges network + secrets
  • ASK_PPTX_AGENT UDF — Generates JWT and calls the agent API
  • CALL_PPTX_AGENT_EXTERNAL — Procedure wrapper for tool binding

Prerequisites

1. Generate RSA key pair: openssl genrsa 2048 | openssl pkcs8 -topk8 -inform PEM -out rsa_key.p8 -nocrypt

2. Create service user with public key (ACCOUNTADMIN)

3. Update the SECRET in the script with your private key

5Integrate with Your Agent

Add the PPTX tool to your existing agent

Layer 5

Add two blocks to your main agent's YAML specification:

  • tool_spec — Declares the tool with type: "generic"
  • tool_resources — Binds to CALL_PPTX_AGENT_EXTERNAL procedure
See README.md for the exact YAML blocks to copy.
